# 指令手册

## 清屏(.cls)

- `.cls`

    **功能**：清空当前窗口输出。

## 重新加载符号文件(.reload)

- `.reload`

    **功能**：为所有已加载的模块重新加载符号。

- `.reload /f`

    **功能**：强制重新加载所有模块的符号，即使它们看起来已经加载了 。这在符号加载有问题时非常有效。

- `.reload /f <module name>`

    **功能**：仅强制重新加载指定模块的符号。注意，模块名需要包含完整的文件名后缀（如 .exe、.dll、.sys）

## 查看与设置符号文件检索路径(.sympath)

- `.sympath`

    **功能**：显示当前已配置的符号路径。

- `.sympath <path1>;<path2>;...`

    **功能**：设置多个符号文件检索路径。

- `.sympath+ <path>`

    **功能**：追加一个新路径到现有路径列表的末尾。

## 列出模块信息(lm)

- `lm`

    **功能**：以标准格式列出所有模块。

    **输出示例**：
    ```
    start             end                 module name
    00007ff6`2c7b0000 00007ff6`2c7b9000   Test     C (private pdb symbols)  d:\develop\code\component\my\xnetwork\test\bin\release\Test.pdb
    00007ffd`51d00000 00007ffd`51d8d000   MSVCP140   (deferred)             
    00007ffd`54b50000 00007ffd`54b6d000   VCRUNTIME140   (deferred)             
    00007ffd`54f30000 00007ffd`54f3c000   VCRUNTIME140_1   (deferred)             
    00007ffd`65520000 00007ffd`65530000   XNetwork C (private pdb symbols)  d:\develop\code\component\my\xnetwork\xnetwork\bin\release\XNetwork.pdb
    00007ffd`67960000 00007ffd`679cc000   MSWSOCK    (export symbols)       C:\Windows\SYSTEM32\MSWSOCK.dll
    00007ffd`683f0000 00007ffd`684f0000   ucrtbase   (export symbols)       C:\Windows\System32\ucrtbase.dll
    00007ffd`684f0000 00007ffd`687e6000   KERNELBASE   (pdb symbols)          d:\develop\symbols\kernelbase.pdb\8DE470CE6A35310DEB2DAA4415DC675E1\kernelbase.pdb
    00007ffd`68fd0000 00007ffd`69092000   KERNEL32   (pdb symbols)          d:\develop\symbols\kernel32.pdb\B18A8862AA051500CA660D171E4963A21\kernel32.pdb
    00007ffd`69760000 00007ffd`69883000   RPCRT4     (deferred)             
    00007ffd`699a0000 00007ffd`69a0b000   WS2_32     (deferred)             
    00007ffd`6ab30000 00007ffd`6ad28000   ntdll      (pdb symbols)          d:\develop\symbols\ntdll.pdb\E84AF4A1E3CDFABC99A41209581EB6861\ntdll.pdb

    ```

    - `start`(起始地址)和`end`(结束地址)

        模块在虚拟内存中的起始地址和结束地址。整个模块的代码和数据就位于`start`到 `end-1`的地址范围内。

        64位地址，通常显示为00007ff6`2c7b0000这样的形式。
        - 高32位`00007ff6`表示地址空间的上半部分(用户空间)
        - 低32位`2c7b0000`是具体偏移地址

    - `module name`(模块名称)

        加载的模块(EXE/DLL)的文件名。
        常见类型：
        - 应用程序EXE
        - 系统DLL(如ntdll、KERNEL32)
        - 运行时库(如MSVCP140、VCRUNTIME140)

    - 符号状态
        指示该模块的调试符号加载状态

        |状态值|含义|说明|
        |-|-|-|
        |(private pdb symbols)|私有PDB符号已加载|成功加载了包含完整类型、函数、局部变量信息的PDB符号文件。这是调试时的理想状态|
        |(export symbols)|导出符号|未找到PDB符号文件，因此调试器仅从模块的导出表中提取了有限的符号信息（主要是函数名）|
        |(pdb symbols)|公共PDB符号|使用公开的符号服务器下载的PDB|
        |(deferred)|延迟加载|模块已加载，但调试器尚未尝试加载其符号。这是正常状态，符号会在需要时（例如设置断点、查看堆栈时）自动加载|
        |(no symbols)|无符号|找不到任何符号信息|

    - 模块路径/PDB路径

        模块文件或符号文件的实际路径。

        - 对于有符号的模块：显示PDB(程序数据库)文件路径

        - 对于无符号/仅导出符号的模块：显示模块文件本身路径

## 显示异常记录详细信息(.exr)

- `.exr -1`
    
    **功能**：用于显示最近发生的异常记录。

    **输出示例**：
    ```
    ExceptionAddress: 00007ffb8021121c (DLL1!ResourceManager::WorkerThreadProc+0x000000000000001c)
    ExceptionCode: c0000005 (Access violation)
    ExceptionFlags: 00000000
    NumberParameters: 2
    Parameter[0]: 0000000000000001
    Parameter[1]: 000001cd7862be70
    Attempt to write to address 000001cd7862be70
    ```

    - `ExceptionAddress`：这是发生异常的确切指令地址，并且如果符号加载成功，会显示这条指令所在的函数名及偏移。

    - `ExceptionCode`：异常代码，用于识别异常类型。

    - `ExceptionFlags`：异常标志。

        - `00000000` 表示这是一个“可继续”的异常。

        - `00000001` 表示这是一个“不可继续”的异常。
    
    - `NumberParameters`：异常附带参数的个数。

    - `Parameter[0]` 和 `Parameter[1]`：异常的具体参数，其含义随异常代码而变化。

    - `最后的描述行`：这是调试器对异常代码和参数进行的友好解读。

## 显示线程函数调用堆栈信息(kv)

>**栈地址**：通常位于用户空间的高地址区域（如`0000002fxxxxxxxx`）
>**代码地址**：位于模块加载基址附近（如`00007ffdxxxxxxxx`）

- `kv`

    **功能**：查看当前线程的函数调用堆栈(`k`)回溯详细(`v`)信息。

    **输出示例**：
    ```
    # Child-SP          RetAddr               : Args to Child                                                           : Call Site
    00 0000008a`86fffba0 00007ffb`84f97374     : 00000000`00000000 00000000`00000000 00000000`00000000 00000000`00000000 : DLL1!ResourceManager::WorkerThreadProc+0x1c
    01 0000008a`86fffbd0 00007ffb`85fbcc91     : 00000000`00000000 00000000`00000000 00000000`00000000 00000000`00000000 : kernel32!BaseThreadInitThunk+0x14
    02 0000008a`86fffc00 00000000`00000000     : 00000000`00000000 00000000`00000000 00000000`00000000 00000000`00000000 : ntdll!RtlUserThreadStart+0x21
    ```

    - **#-栈帧编号**

        从00开始，00是当前帧。数字越大，调用层次越深（越早的调用者）。

    - **Child-SP(子栈帧基址)**

        栈内存布局（从高地址向低地址增长）：

        ```
        高地址
        +---------------------+
        | 父函数的栈帧        |
        | 局部变量等          |
        +---------------------+ <-- 父函数的RSP在执行call指令前的位置
        | 返回地址 (RetAddr)  | (8字节)
        +---------------------+ <-- 子函数的RSP初始位置
        | 参数区域            | (可选，x64前4个参数通常用寄存器传递)
        +---------------------+
        | 子函数的栈帧        |
        | 局部变量等          |
        +---------------------+
        低地址
        ```

        - 函数的所有局部变量都存放在这个栈帧地址之上（向低地址方向）。

        - 在x64架构中，当发生call指令时：

            - CPU自动将返回地址（RetAddr）压入栈顶

            - 栈指针（RSP）自动减少8字节（因为栈向低地址增长）

            - 子函数开始执行，建立自己的栈帧（Child-SP）

        - 栈帧包含内容：

            - 返回地址（8字节）

            - 函数参数（x64前4个参数通常用寄存器，其余压栈）

            - 局部变量

            - 保存的非易失性寄存器

    - **RetAddr**

        返回地址(当前函数执行完毕后应返回的地址)，此处的地址为**代码地址**。
        此处`DLL1!ResourceManager::WorkerThreadProc`函数的返回地址为`00007ff91d027374`，其就等于`KERNEL32!BaseThreadInitThunk+0x14`，其中`KERNEL32!BaseThreadInitThunk`表示函数的基代码地址，即对应函数第一条汇编指令的代码地址。

    - **Args to Child**

        传递给被调用函数的参数(最多显示4个参数)。

    - **Call Site**

        调用位置(函数名+偏移量)，代码地址。

- `~* kv`

    **功能**：查列出所有线程（`~*`）的详细信息，并显示每个线程的调用堆栈（`k`）和关键参数（`v`）。

    
## 根据内存地址，查找符号(ln)

- `ln <Address>`

    **功能**：根据给定的内存地址，查找并显示离该地址最近的一个或多个符号（如函数名、全局变量名）。

    **输出示例**：

    ```
    ln 00007ff9`1d027374

    (00007ff9`1d027360)   kernel32!BaseThreadInitThunk+0x14   |  (00007ff9`1d0273ac)   kernel32!BasepInitializeTermsrvFpns
    ```

    - `(00007ff91d027360)`为符号(函数)`kernel32!BaseThreadInitThunk`的起始内存地址

    - `kernel32!BaseThreadInitThunk+0x14`表示查询地址位于 `kernel32!BaseThreadInitThunk`函数内，距离函数入口有`0x14`个字节的偏移

    - `(00007ff91d0273ac)   kernel32!BasepInitializeTermsrvFpns`表示在地址空间中，紧随当前符号之后的下一个符号是`kernel32!BasepInitializeTermsrvFpns`，它的起始内存地址是`00007ff91d0273ac`。

## 恢复中断继续执行(g)

- `g`

    **功能**：让程序从当前的指令地址开始自由运行。

- `g <Address>`

    **功能**：运行到指定地址：在地址上设置一个一次性断点。程序开始运行，当执行流到达该地址时，会自动中断。


## 设置断点(bp)

- `bp <Address>`

    **功能**：在指定的内存地址设置断点。

    示例：`bp MyApp!MyClass::MyMethod+0x24`：在 MyClass::MyMethod 函数入口偏移 0x24 字节的位置设置断点（即函数内部的具体位置）。

## 步过(p)

- `p`

    **功能**：执行单条指令或源代码行，但如果遇到函数调用，则将整个函数作为一步执行完毕（即“步过”），不会进入函数内部。

- `Pa <Address>`

    **功能**：步过到指定地址：连续执行（步过模式），直到执行流到达指定地址才停下来。

- `pct`

    **功能**：步过直到下一个调用或返回指令：连续执行（步过模式），直到遇到下一个 `call`、`ret` 或 `retf` 程序指令。

## 步入(t)

- `t`

    **功能**：执行单条指令或源代码行，但如果遇到函数调用指令（如 call），则会“步入”该函数内部，停在被调函数的第一条指令上。

- `ta <Address>`

    **功能**：步到指定地址：连续执行（步入模式），直到执行流到达指定地址才停下来。

- `tct`

    **功能**：步到下一个调用或返回指令：连续执行（步入模式），直到遇到下一个 `call`、`ret` 或 `retf` 指令。


## 创建内存转储文件(.dump)

命令示例	参数选项含义	文件大小	包含的信息与适用场景
.dump /ma C：\dump\full.dmp	/ma （MiniDumpWithFullMemory）	大	这是一个最全面的选项。它包含了进程的完整内存内容（包括所有模块的内存、句柄、线程信息等）。如果你需要分析复杂的内存问题、变量值或完整堆数据，使用这个选项。这是最常用、最推荐的选项。
.dump /m C：\dump\mini.dmp	/m （MiniDumpNormal）	小	生成一个标准的小型转储。它只包含最基本的信息，如线程堆栈、模块列表和部分内存。文件体积小，便于传输，但可能无法查看全局变量或特定内存区域的值。
.dump /f C：\dump\fulllegacy.dmp	/f （Full Dump - 旧式）	大	生成一个传统的完整转储。虽然也包含完整内存，但现代调试中通常被功能更丰富的/ma替代。
.dump /mdi C：\dump\minidump.dmp	/mdi （添加更多信息）	中等	在标准小转储的基础上，添加了更多信息，比如可执行映像的镜像。这对于调试本地代码崩溃非常有用，因为它允许查看正在执行的代码指令，即使没有完整的二进制文件。
.dump /mhiptu C：\dump\special.dmp	/mhiptu	可变	这是一组组合选项，用于生成包含特定内存区域（如句柄、环境变量、线程时间等）的转储文件，常用于满足特定的调试需求。
